Start

First, register an OAuth Apps application in Github's personal settings-Develop settings, and fill in the relevant information in the configuration interface according to your own blog.
Or set directly at: https://github.com/settings/applications/new

Among them, the Application name can be filled in according to your own ideas, which is the application name of this OAuth Apps.
Homepage URL is your own homepage address. If you have not bound a domain name, fill in: username.github.io. If you have bound your own domain name, fill in your own domain name address.
Application description Fill in the application description according to your actual situation.
Authorization callback URL and Homepage URL should be the same.

After the setting is complete, a Client ID and Client Secret will be generated, which are copied or recorded here for use in the next configuration.

Configuration in Gridea

The configuration in Gridea is relatively simple, you only need to fill in the information corresponding to the OAuth Apps you just applied for.

Among them, the warehouse used to store comments can be changed. You can use an independent warehouse for storage, or you can use your own Github Pages warehouse for storage. Owner is your own username.
After filling in, click Save and then sync.
If your repo has not enabled the issue function, then you need to enable the issue in the warehouse settings where comments are stored, so that it can be used.

The pit I stepped on in the configuration

After the start configuration is complete, although the comment interface is displayed at the end of the article, as long as I click to log in to comment, it will jump to the home page. I checked the information on the Internet and found that there is a problem with the configuration of the Authorization callback URL and Homepage URL in the OAuth Apps configuration page. I bound my own domain name on the blog: wherelse.cc, I filled in the default link of my github pages: wherelse.github.io, which caused problems in comments.
